# 混合 RAG 架构方案

## 文档信息

- **版本**: 1.0
- **创建日期**: 2026-03-05
- **目标场景**: 企业内部知识库
- **文档规模**: 100 万（一期），可扩展至 1000 万
- **延迟要求**: 秒级（2-6 秒）
- **精度要求**: 错误率 <10%，需要溯源

---

## 核心架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户查询                                  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  存储层 (已有基础设施)                                           │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌────────┐ │
│  │ PostgreSQL  │  │Elasticsearch│  │   Neo4j     │  │ Redis  │ │
│  │ + pgvector  │  │  全文检索   │  │  溯源图谱   │  │  缓存  │ │
│  │  向量存储   │  │             │  │             │  │        │ │
│  └─────────────┘  └─────────────┘  └─────────────┘  └────────┘ │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  检索层 (需开发)                                                 │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐             │
│  │  查询改写   │→ │ 融合召回    │→ │ 轻量重排序  │             │
│  │             │  │ (向量+ES)   │  │  (规则)     │             │
│  └─────────────┘  └─────────────┘  └─────────────┘             │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  生成层                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐             │
│  │ Neo4j 溯源  │→ │ 上下文组装  │→ │ LLM 生成    │             │
│  │ 图谱查询    │  │             │  │ + 引用标注  │             │
│  └─────────────┘  └─────────────┘  └─────────────┘             │
└─────────────────────────────────────────────────────────────────┘
```

## 技术选型

### 存储层

| 组件 | 用途 | 选型理由 |
|------|------|----------|
| PostgreSQL + pgvector | 向量存储 + 元数据 | 已有基础设施，百万级支持，商业友好 |
| Elasticsearch | 全文检索 (BM25) | 已有基础设施，关键词匹配 |
| Neo4j | 文档关系 + 溯源图谱 | 已有基础设施，支持多跳引用 |
| Redis | 查询缓存 + 会话 | 已有基础设施，低延迟 |

### 检索层

| 组件 | 用途 | 实现方式 |
|------|------|----------|
| 查询改写 | 查询扩展/标准化 | 规则 + LLM |
| 融合召回 | 向量 + ES 结果融合 | RRF 算法 |
| 重排序 | 结果精排 | 规则（一期）/ LLM（二期） |

### 生成层

| 组件 | 用途 | 选型 |
|------|------|------|
| LLM | 答案生成 | OpenAI GPT-4o / Anthropic Claude |
| 引用标注 | 溯源 | 基于 Neo4j 图谱 |

## 性能指标

| 阶段 | 延迟 | 备注 |
|------|------|------|
| 缓存检查 | <5ms | Redis |
| 向量检索 | 30-50ms | pgvector HNSW |
| ES 检索 | 20-40ms | 现有 ES |
| RRF 融合 | <10ms | 内存计算 |
| 重排序 | 0-500ms | 规则=0ms, LLM=500ms |
| Neo4j 溯源 | 100-200ms | 图查询 |
| LLM 生成 | 2-5s | OpenAI/Anthropic |
| **总计** | **2.5-6s** | 符合秒级要求 |

## 风险点

1. **pgvector 性能**：百万级 HNSW 索引构建时间长（数小时），需离线构建
2. **Embedding 成本**：100 万文档 × OpenAI embedding ≈ $1000-2000（一次性）
3. **Neo4j 复杂度**：图谱设计需要业务理解，建议简化起步
4. **ES 同步**：文档更新时需同步 ES 和 pgvector，注意一致性
