# 技术博客 / 文章研究方法（通用）

本方法用于研究单篇技术博客/文章，并将结果沉淀到知识库：`research/<domain>/<blog-name>.md`。

目标：**结论优先、证据可追溯、可复用**（后续同领域文章可以横向对比）。

> 约束：
> - 文档只给建议（优势/劣势/适用场景/风险/证据），不做“用/不用”的最终决策。
> - 若文中引用代码/实现：必须提供**源文件+行号**链接（GitHub permalink 优先）。

---

## 1. 研究流程（建议步骤）

1) **登记元信息**：作者/发布时间/来源链接/**自动生成 domain（可由你后续调整/移动）**/关键词
2) **一句话结论（TL;DR）**：用 3~7 条要点写清文章最值得吸收的东西
3) **问题定义**：文章要解决的核心问题是什么？适用前提是什么？
4) **方法/方案拆解**：
   - 核心概念/术语表
   - 关键步骤（流程图/要点）
   - 关键参数（默认值、取值范围、调参逻辑）
5) **证据与复现**：
   - 文章给出的实验/benchmark/案例是否充分
   - 能否复现（需要哪些依赖/数据/配置）
6) **建议与风险**：
   - 优势
   - 劣势/限制
   - 适用场景
   - 风险与成本（性能、维护、供应商锁定、中文效果等）
7) **延伸检索（强烈建议）**：
   - 围绕文章主题，用 web search / 其他来源补充 3~10 个“类似方案/相反观点/落地实践/评测论文”。
   - 对每个延伸条目，必须补齐：
     - 一句话总结（它在讲什么 / 提供什么方法）
     - 与主文章的区别（视角/切入点/适用前提）
     - 优劣势与取舍（对你工程化落地的启发）
   - 目标：让你通过一篇文章拿到更完整的方案空间。
8) **落地动作（可选）**：
   - 我们可直接复用的清单（设计/参数/流程/代码片段）
   - 需要验证的假设（下一步实验）
9) **回填索引**：在根目录 `README.md` 的 Research 表格新增一条记录（链接到报告）。
10) **登记去重**：在 `research/FEEDS.md` 的“已研究文章”表格登记该文章（后续不再推送）。

---

## 2. 输出文件模板（research/<domain>/<blog-name>.md）

建议固定结构如下（结论优先）：

- 标题 + 来源链接（URL）
- 元信息（作者/文章发布日期/报告生成日期/domain/关键词）
- TL;DR（3~7 条）
- 文章解决的问题（Problem)
- 方案拆解（Approach)
- 关键细节（参数/流程/边界条件）
- 优势
- 劣势/限制
- 适用场景
- 风险与成本
- 可复用清单（我们能直接抄的点）
- **原文要点摘录与中文释义（短引用对照）**
- 图/非文本内容（图片等，保持顺序，给出链接与说明）
- **延伸检索：类似方案/思路（跨来源补充）**（默认粒度：每条一个小节，尽量控制在“一屏内”；必须包含：总结 + 与主文区别 + 优劣势/取舍）
- 参考与证据（链接）

---

## 3. 引用规范（强制）

### 3.1 文章引用
- 必须提供文章原链接（必要时也记录归档链接）。
- 为了便于不回看原文：建议在“原文要点摘录与中文释义”章节中，按段落/要点给出**短引用（每段不超过 90 字符）** + 中文释义。
  - 注：出于版权与可分享性考虑，知识库默认不收录整篇文章的逐段全文翻译；如你确实需要全文对照，请你提供文章全文文本（你拥有/可用的版本），我再按你提供的文本生成对照译文。

### 3.2 代码引用
- 如果文章引用/展示了源码：
  - 优先使用 GitHub permalink（含 commit hash）并带行号：
    - `https://github.com/<owner>/<repo>/blob/<commit>/<path>#L10-L42`
  - 在代码块旁标注：
    - `Source: <url>#Lx-Ly`

---

## 4. 命名约定（blog-name）

- 建议使用：`YYYY-MM-DD-<slug>` 或 `<site>-<short-title>`
- 规则：全小写、用 `-` 分隔、避免中文与空格。
